# A Truly Prompt-Driven UI
To move beyond “button click” metaphors and toward a natural-language, prompt-only interface (typed or spoken), we introduce a set of **Prompt Primitives**—reusable, composable units that guide the user solely by language. These primitives knit together with **Practices**, **Insights**, and **Moments** to create a dynamic, conversational affordance layer.
## 1. Prompt Primitives
Each primitive is a pattern of language prompting that can be rendered in text or spoken cues.
1. **Open Prompt**
   A free-form invitation, often backed by a hint pattern.
   Example:
   “What would you like to do next?
   • Generate a report
   • Validate input schema
   • Explain protocol details”
2. **Slot Prompt**
   A placeholder request for a specific piece of information, with dynamic defaults from Insights.
   Template:
   “Please provide the {𝑋} (default: {𝑥₀}).”
   Example:
   “Please provide the report format (default: XLSX).”
3. **Choice Prompt**
   A closed list of options, enumerated in text.
   Template:
   “Choose one of: {A}, {B}, {C}.”
   Example:
   “Choose one of: CSV, XLSX, JSON.”
4. **Confirm Prompt**
   A yes/no confirmation or acceptance check.
   Template:
   “Do you want to proceed with {action}? (yes/no)”
5. **Feedback Prompt**
   Captures user reaction to a previous moment.
   Template:
   “Was this output helpful? (yes/no/edit)”

Here’s a quick comparison of **state-driven** versus **free-form** prompt interfaces, and a hybrid that tries to get the best of both worlds.
## 1. State-Driven Interfaces
### Pros
• **Guidance & Predictability**
Users always know what kind of response is expected next.
• **Error Reduction**

Every prompt→reply is a known event/transition, simplifying Moment recording.

Flows are finite state machines—you can write deterministic tests.

• **Rigidity**
Hard to handle out-of-sequence requests or complex, multi-intent utterances.
• **Maintenance Overhead**
Every new branch or exception requires editing the state net.
• **User Frustration**
Power users may feel boxed in by “you must do X before Y.”
## 2. Free-Form Interfaces
### Pros
• **Flexibility**
Users can type or say anything at any time—natural conversational style.
• **Lower Authoring Cost**
No need to encode every branch in a state machine.
• **LLM-Centered**
Let the LLM interpret intent and slot-fill on the fly.
### Cons
• **Unpredictability**
User utterances may be ambiguous or out-of-scope.
• **Validation Burden**
Must use NLU/LLM post-processing to extract intent, slots, and detect errors.
• **Moment Mapping**
Harder to map arbitrary text to structured events, complicating analytics.

2. **LLM Intent Parsing**

3. **Slot-Fill & Back-Fill**
   If the user says “Create role Admin,” you detect intent=`createRole`, slot `roleName=Admin`, and skip ahead in any underlying state net.
4. **Fallback Prompts**

### Tradeoffs of Hybrid
• **Complexity** — you need an NLU/slot-filling layer.
• **Flexibility** — users drive the pace, but you still guide missing pieces.
• **Resilience** — robust to both novice and expert users.
• **Analytics** — you map free-form inputs into structured intents/slots, then record Moments as before.

      slots[field] = await promptText(ask);
    }
  }

  let obj = await AClass.create(slots);

  // 4. Confirmation & next steps
  await promptText(
    overrides.confirmation
      || `Created ${entityName} “${slots.name}”. What would you like to do next?`
  );
  return obj;
}
